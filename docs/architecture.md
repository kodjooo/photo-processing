Архитектура MVP сервиса автоматической обработки фотографий

1. Границы MVP
- На вход принимается только публичная ссылка Яндекс Диска на ZIP-архив.
- Запуск и эксплуатация проекта выполняются только через Docker Desktop и `docker compose`.
- На выходе MVP формирует ZIP-архив с обработанными изображениями и загружает его обратно в Яндекс Диск.
- Поддерживаются изображения форматов JPG, JPEG, PNG и WEBP.
- RAW и любые неподдерживаемые файлы пропускаются с фиксацией причины в отчете.

2. Состав сервисов
- `app` — контейнер с HTTP API и общими модулями приложения.
- `bot` — контейнер Telegram-бота, который принимает команды пользователей и обращается к сервисному слою.
- `worker` — контейнер фонового обработчика, который забирает задачи из Redis и выполняет полный пайплайн.
- `postgres` — база данных со статусами задач, отчетами и техническими событиями.
- `redis` — брокер очереди задач и вспомогательное хранилище для фонового выполнения.

3. Выбранный стек
- Python 3.12.
- FastAPI для HTTP API и healthcheck.
- Aiogram для Telegram-бота.
- SQLAlchemy + PostgreSQL для хранения задач и отчетов.
- Redis как очередь задач и канал координации фонового выполнения.
- Pillow как базовая библиотека обработки изображений.
- `docker compose` как единственный способ локального запуска.

4. Логическая схема
- Telegram-пользователь отправляет команду или ссылку боту.
- Если пользователь присылает только ссылку, бот сначала показывает inline-кнопки выбора пресета: три режима локальной проработки и три режима общей проработки.
- После выбора пресета бот создает задачу через сервисный слой.
- Job Manager сохраняет запись в PostgreSQL и публикует `job_id` в Redis.
- Worker забирает задачу, скачивает архив, распаковывает его, обрабатывает изображения, формирует ZIP и загружает результат в Яндекс Диск.
- После завершения worker обновляет статус, отчет и итоговую ссылку.
- Бот по команде `/status` или `/last` читает данные из PostgreSQL и возвращает пользователю актуальную информацию.

5. Директории временных файлов
- Рабочая директория задачи: `/tmp/jobs/{job_id}`.
- Входной архив: `/tmp/jobs/{job_id}/input/archive.zip`.
- Распакованные файлы: `/tmp/jobs/{job_id}/input/unpacked`.
- Результат обработки: `/tmp/jobs/{job_id}/output`.
- Финальный архив: `/tmp/jobs/{job_id}/result/result.zip`.

6. Статусы задач
- `created`
- `validating`
- `queued`
- `downloading`
- `unpacking`
- `processing`
- `packaging`
- `uploading`
- `completed`
- `failed`
- `cancelled`

7. Основные сущности данных
- `jobs` — основная запись задачи с `job_id`, пользователем, исходным Telegram chat id, ссылкой, пресетом, статусом, статистикой и ссылкой на результат.
- `job_events` — история переходов статусов и технические сообщения по задаче.
- `job_files` — список обработанных и пропущенных файлов с итогом и причиной.

8. Конфигурация
- Все параметры читаются только из переменных окружения.
- `.env.example` хранит шаблон переменных.
- `.env` используется только локально и не должен попадать в Git.
- В `.env` должны быть комментарии о назначении ключей и месте получения доступа.

9. Правила упрощения
- В MVP используется один основной сценарий выгрузки результата: ZIP-архив.
- Бот и API используют общий код приложения, чтобы не дублировать бизнес-логику.
- Очередь реализуется поверх Redis без тяжелых оркестраторов задач.

10. Реализованная структура приложения
- `app/main.py` — единая точка входа с командами `api`, `bot`, `worker`, `init-db`.
- `app/api.py` — HTTP API с healthcheck и операциями по задачам.
- `app/bot.py` — Telegram-бот на `aiogram` с командами `/start`, `/help`, `/process`, `/status`, `/presets`, `/logo`, `/cancel`, `/last` и inline-кнопками выбора локального или общего пресета.
- `app/repositories/jobs.py` — слой доступа к данным задач, событий и файлов.
- `app/services/job_manager.py` — создание задач, отмена, чтение статусов и постановка в очередь.
- `app/services/job_processor.py` — полный пайплайн обработки фоновой задачи.
- `app/services/yandex_disk.py` — работа с публичными ссылками и выгрузкой результата в Яндекс Диск.
- `app/services/storage.py` — временные директории, безопасная распаковка и упаковка результата.
- `app/processing/images.py` — расчет метрик, улучшение изображения и наложение логотипов.

11. HTTP API
- `GET /health` — проверка доступности сервиса.
- `POST /jobs` — создание задачи.
- `GET /jobs/{job_id}` — получение статуса и отчета по задаче.
- `POST /jobs/{job_id}/cancel` — отмена задачи.
- `GET /users/{telegram_user_id}/jobs/last` — последняя задача пользователя.

12. Модель хранения данных
- Таблица `jobs` хранит исходную ссылку, статус, пресет, `telegram_user_id`, `telegram_chat_id`, времена начала и завершения, счетчики файлов, текст ошибки и ссылку на результат.
- Поле `telegram_user_id` хранится как `BIGINT`, потому что реальные Telegram ID не помещаются в `int32`.
- Таблица `job_events` хранит историю смены статусов и сообщения прогресса.
- Таблица `job_files` хранит результат по каждому файлу: обработан, пропущен, ошибка и причина.
- Инициализация схемы выполняется отдельной командой `init-db` в контейнере `migrate`.

13. Очередь и выполнение задач
- Постановка в очередь идет через Redis-список `photo_processing:jobs`.
- Worker использует `BRPOP` и обрабатывает задачи последовательно с ограничением `WORKER_CONCURRENCY`.
- Во время обработки worker пишет события прогресса по файлам в `job_events`.
- Отмена задачи проверяется между ключевыми этапами и перед обработкой каждого файла.
- После завершения или ошибки worker пытается отправить Telegram-уведомление обратно в исходный чат по `telegram_chat_id`.
- Ошибки Telegram-уведомлений не должны переводить уже завершенную задачу в `failed` и не должны останавливать `worker`.

14. Пайплайн задачи
- Валидация публичной ссылки Яндекс Диска.
- Получение метаданных ресурса и контроль лимита по размеру.
- Потоковое скачивание ZIP-архива во временную директорию.
- Безопасная распаковка с защитой от path traversal.
- Сканирование файлов, фильтрация поддерживаемых изображений и сбор причин пропуска.
- Последовательная обработка изображений с настраиваемой внутренней параллельностью.
- Упаковка выходного каталога в `result.zip`.
- Автоматическое создание целевой директории на Яндекс Диске при отсутствии.
- Загрузка результата в Яндекс Диск и сохранение публичной ссылки.

15. Обработка изображений
- Используется `Pillow`.
- Для RAW-форматов `.CR2` и `.ARW` используется `rawpy` с LibRaw-совместимой postprocess-конвертацией в RGB.
- Результаты обработки RAW сохраняются как `.jpg`, а не обратно в `.cr2`/`.arw`.
- Перед обработкой применяется `EXIF transpose` и приведение к RGB.
- Вычисляются базовые метрики: яркость, контраст, резкость, доля темных и пересвеченных областей.
- На основе метрик и пресета подбираются коэффициенты глобальной и локальной коррекции.
- Пайплайн использует деликатную локальную обработку теней и светов: темные зоны подтягиваются умеренно, а пересветы сдерживаются без выраженного HDR-эффекта.
- Локальный контраст усиливается через мягкий detail-layer поверх размытой базовой версии кадра, чтобы добавлять объем без грубой фактуры и ореолов.
- Цвет усиливается через мягкую vibrance-коррекцию: менее насыщенные участки усиливаются сильнее уже насыщенных, а skin-tone диапазон ограничивается от чрезмерной сочности.
- Финальная резкость собирается через ослабленный `UnsharpMask` и мягкую глобальную резкость, чтобы избежать жесткого перешарпа на шумных снимках.
- Поддерживаются две семьи пресетов:
- локальная проработка: `natural`, `balanced`, `strong`;
- общая проработка: `global_natural`, `global_balanced`, `global_strong`;
- `default` работает как алиас `balanced`.
- Локальные пресеты используют адаптивную теневую, световую и detail-layer коррекцию.
- Глобальные пресеты ограничиваются общей коррекцией яркости, контраста, цвета и резкости без локальной проработки зон.
- Если ссылка отправлена без пресета, выбор режима выполняется через inline-кнопки в Telegram.
- Логотипы масштабируются адаптивно: `15%` ширины кадра для горизонтальных изображений и `25%` ширины кадра для вертикальных.
- На результат накладываются два логотипа с повышенной непрозрачностью `0.55` в нижних углах.

18. Поддерживаемые входные форматы
- Контейнер задачи: только ZIP-архив.
- Растровые изображения: `JPG`, `JPEG`, `PNG`, `WEBP`.
- RAW для MVP+: `CR2`, `ARW`.
- Остальные файлы продолжают фиксироваться как пропущенные в отчете.
- Служебные файлы macOS из `__MACOSX` и `._*` автоматически пропускаются.

16. Docker-only схема запуска
- Все сервисы запускаются только через `docker compose`.
- Основной файл оркестрации: `docker-compose.yml`.
- Сервисы `api`, `bot`, `worker` используют общий образ из `Dockerfile`.
- `postgres`, `redis` и общее временное хранилище подключены через docker volumes.
- Healthcheck настроен для `postgres`, `redis` и `api`.
- В образе задается `PYTHONPATH=/app`, чтобы контейнерные тесты и все entrypoint-команды видели пакет `app` одинаково.

17. Тестовое покрытие
- `tests/test_validators.py` — валидация ссылок Яндекс Диска.
- `tests/test_storage.py` — безопасная распаковка и фильтрация файлов.
- `tests/test_image_processor.py` — базовая обработка изображения и сохранение результата.
- `tests/test_api.py` — smoke-проверка healthcheck API.
- `tests/test_bot.py` — проверка inline-кнопок выбора пресета.
