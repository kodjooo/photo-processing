ТЕХНИЧЕСКОЕ ЗАДАНИЕ
Сервис автоматической обработки фотографий через Telegram-бот с использованием Яндекс Диска

1. Цель проекта

Разработать сервис, который позволяет внешнему фотографу или сотруднику:
1) загрузить архив с фотографиями на Яндекс Диск;
2) открыть доступ по ссылке;
3) отправить ссылку в Telegram-бот;
4) автоматически скачать архив;
5) обработать изображения по заданному пайплайну;
6) добавить на каждое изображение два полупрозрачных логотипа в нижних углах;
7) сформировать результат;
8) загрузить результат обратно на Яндекс Диск;
9) вернуть пользователю ссылку на готовый архив или папку через Telegram-бот.

Система должна быть рассчитана на обработку крупных архивов, например:
- до 5 ГБ на один архив;
- ориентировочно до 150 изображений в одной задаче.

2. Общая логика работы

Пользовательский сценарий:
1. Фотограф загружает ZIP-архив с фотографиями на Яндекс Диск.
2. Фотограф включает доступ по ссылке.
3. Фотограф отправляет ссылку в Telegram-бот.
4. При необходимости фотограф указывает режим обработки.
5. Бот принимает задачу, валидирует ссылку и ставит задачу в очередь.
6. Сервер скачивает архив по ссылке с Яндекс Диска потоково на локальное временное хранилище.
7. Архив распаковывается.
8. Каждое изображение обрабатывается автоматически:
   - нормализация;
   - анализ метрик;
   - адаптивное улучшение через фильтры;
   - добавление логотипов в нижний левый и нижний правый углы.
9. Результат сохраняется в отдельную выходную папку.
10. После завершения обработки результат:
   - либо собирается в ZIP-архив;
   - либо загружается на Яндекс Диск как папка;
   - предпочтительно поддержать оба варианта, с одним выбранным вариантом по умолчанию.
11. Бот возвращает ссылку на результат и краткий отчет по обработке.

3. Основные требования к функциональности

3.1. Telegram-бот
Бот должен уметь:
- принимать ссылку на публичный архив на Яндекс Диске;
- принимать команды для запуска обработки;
- показывать статус задачи;
- возвращать ссылку на результат;
- возвращать отчет об успешных и пропущенных файлах;
- при ошибке возвращать понятное сообщение.

Поддерживаемые команды:
- /start — приветствие и краткая инструкция;
- /help — инструкция по формату ссылки, ограничениям и режимам обработки;
- /process <ссылка> — создать задачу на обработку;
- /status <job_id> — показать текущий статус задачи;
- /presets — показать доступные режимы обработки;
- /logo — краткая информация о требованиях к логотипам;
- /cancel <job_id> — отмена задачи, если она еще не завершена;
- /last — информация о последней задаче пользователя.

Дополнительно допускается режим без команды:
- пользователь просто отправляет ссылку;
- бот распознает ссылку и запускает обработку по умолчанию либо предлагает формат короткого выбора режима.

3.2. Работа со ссылками Яндекс Диска
Система должна:
- принимать только ссылки Яндекс Диска на публично доступные файлы;
- проверять доступность ссылки;
- определять, что по ссылке доступен именно ZIP-архив;
- корректно обрабатывать ошибки доступа;
- корректно сообщать пользователю, если файл недоступен, удален или не является архивом.

3.3. Формат входных данных
На MVP предусмотреть:
- входной формат: ZIP-архив;
- внутри архива — изображения в форматах JPG, JPEG, PNG, WEBP;
- вложенные папки допускаются;
- сохранять структуру вложенных папок в результате желательно;
- неподдерживаемые файлы игнорировать и отражать в отчете.

RAW-форматы на MVP не поддерживать, если не будет отдельного требования.
Если встречаются RAW, система:
- не пытается их обрабатывать;
- фиксирует их в отчете как пропущенные.

3.4. Ограничения входных данных
Предусмотреть следующие ограничения:
- максимальный размер архива: 5 ГБ;
- максимальное количество файлов в архиве: 300;
- максимальное количество изображений для обработки в одной задаче: 200 на MVP;
- если лимит превышен, задача должна отклоняться до начала обработки с понятным сообщением.

4. Архитектура решения

Рекомендуемая архитектура:
1. Telegram Bot
2. Backend API / Job Manager
3. Очередь задач
4. Worker обработки
5. Временное файловое хранилище на сервере
6. Интеграция с Яндекс Диском
7. База данных для статусов, логов и метаданных задач

Логическая схема:
Telegram Bot
→ API / Job Manager
→ Queue
→ Worker
→ Temp Storage
→ Yandex Disk Upload
→ Bot Response

4.1. Telegram Bot
Функции:
- прием ссылок;
- прием параметров запуска;
- выдача статуса;
- выдача результата;
- обработка ошибок;
- уведомление о завершении задачи.

4.2. Job Manager
Функции:
- создать запись о задаче;
- присвоить job_id;
- сохранить параметры запуска;
- поставить задачу в очередь;
- обновлять статус задачи.

4.3. Queue
Для MVP допустимо использовать:
- Redis + очередь;
или
- простую очередь в базе данных, если важна простота.

Предпочтительно:
- Redis + воркер.

4.4. Worker
Функции:
- скачать архив;
- распаковать;
- обработать изображения;
- наложить логотипы;
- сформировать результат;
- загрузить результат на Яндекс Диск;
- обновить статус и отчет.

4.5. База данных
Хранить:
- job_id;
- user_id Telegram;
- исходную ссылку;
- режим обработки;
- статус;
- время создания;
- время старта;
- время окончания;
- число найденных файлов;
- число обработанных файлов;
- число пропущенных файлов;
- текст ошибки при наличии;
- итоговую ссылку на результат;
- технические логи.

5. Статусы задач

Поддержать статусы:
- created
- validating
- queued
- downloading
- unpacking
- processing
- packaging
- uploading
- completed
- failed
- cancelled

Пользователь должен иметь возможность получить текущий статус через /status.

6. Требования к обработке файлов

6.1. Скачивание
Скачивание архива должно происходить потоково, без загрузки целиком в оперативную память.

Архив сохраняется во временную папку вида:
- /tmp/jobs/{job_id}/input/archive.zip

6.2. Распаковка
Архив распаковывается в:
- /tmp/jobs/{job_id}/input/unpacked/

Результаты сохраняются в:
- /tmp/jobs/{job_id}/output/

6.3. Обработка изображений
Изображения должны обрабатываться по одному файлу или малыми партиями.
Нельзя загружать все изображения из архива одновременно в RAM.

Рекомендуемая параллельность:
- 2–6 воркеров обработки файлов внутри одной задачи;
- значение должно быть настраиваемым через конфиг.

7. Алгоритм обработки изображения

7.1. Общий принцип
Обработка должна быть аккуратной, без генеративного изменения содержания.
Задача сервиса:
- немного улучшить изображение;
- сделать его визуально более ровным и сочным;
- не менять лица;
- не изменять объект съемки;
- не создавать новые детали;
- не перерисовывать изображение.

Использовать классический adaptive enhancement pipeline на основе фильтров, а не генеративную нейросеть.

7.2. Этапы обработки
Для каждого изображения:

1. Нормализация:
- чтение файла;
- коррекция EXIF-ориентации;
- приведение к рабочему цветовому пространству;
- валидация формата;
- подготовка к обработке.

2. Анализ метрик изображения:
- средняя яркость;
- гистограмма яркости;
- контраст;
- локальный контраст;
- оценка шумности;
- оценка резкости;
- оценка цветового сдвига;
- доля пересвеченных участков;
- доля слишком темных участков.

3. Подбор силы фильтров:
На основе метрик система автоматически подбирает мягкие параметры обработки.

4. Применение фильтров:
- мягкая коррекция white balance;
- мягкая коррекция яркости / gamma;
- мягкая коррекция контраста;
- локальное улучшение контраста;
- легкое шумоподавление при необходимости;
- легкая резкость;
- очень умеренная коррекция насыщенности.

5. Добавление логотипов:
- левый нижний угол;
- правый нижний угол.

6. Сохранение результата:
- сохранить в результирующем формате;
- сохранить структуру папок, если это возможно.

7.3. Ограничения на силу обработки
Для предотвращения порчи изображения ввести защитные ограничения:
- яркость не повышать агрессивно;
- контраст увеличивать умеренно;
- насыщенность не делать чрезмерной;
- не применять сильный шарпинг;
- при высоком шуме снижать резкость;
- при сильных пересветах не поднимать общую экспозицию.

Параметры должны быть вынесены в конфиг и настраиваться без изменения кода.

8. Требования к добавлению логотипов

8.1. Общая логика
На каждое изображение необходимо добавлять два полупрозрачных логотипа:
- один в нижний левый угол;
- один в нижний правый угол.

8.2. Источники логотипов
Система должна поддерживать:
- левый логотип — отдельный файл;
- правый логотип — отдельный файл.

Формат логотипов:
- PNG с прозрачным фоном;
- желательно предусмотреть поддержку SVG как дополнительную опцию, но на MVP достаточно PNG.

8.3. Настраиваемые параметры логотипов
Параметры должны задаваться в конфиге:
- путь к левому логотипу;
- путь к правому логотипу;
- прозрачность;
- масштаб относительно ширины изображения;
- максимальная ширина логотипа;
- максимальная высота логотипа;
- отступ слева;
- отступ справа;
- отступ снизу;
- режим ресайза;
- правило сохранения пропорций.

8.4. Требования к визуальному размещению
Логотипы должны:
- располагаться внутри изображения;
- не выходить за границы кадра;
- сохранять пропорции;
- быть полупрозрачными;
- не перекрывать слишком большую площадь;
- не выглядеть слишком маленькими на больших кадрах;
- не быть слишком большими на маленьких кадрах.

Рекомендуемый принцип масштабирования:
- ширина каждого логотипа = 8–15% от ширины изображения;
- при этом действуют минимальный и максимальный пределы;
- итоговые значения задаются в конфиге.

8.5. Прозрачность
Прозрачность логотипов должна быть настраиваемой.
Рекомендуемый диапазон:
- 20–45% непрозрачности поверх изображения, в зависимости от исходника и дизайна логотипа.

8.6. Безопасные отступы
Рекомендуемые отступы:
- 2–4% от ширины изображения по горизонтали;
- 2–4% от высоты изображения снизу;
- точные значения задаются в конфиге.

8.7. Маленькие изображения
Если изображение слишком маленькое для корректного размещения логотипов, система должна:
- либо уменьшить логотипы до допустимого минимума;
- либо пропустить наложение логотипов и зафиксировать это в отчете;
- конкретное поведение должно задаваться конфигом.

8.8. Ошибки логотипов
Если файлы логотипов отсутствуют или повреждены:
- задача не должна начинать обработку;
- система должна сразу вернуть ошибку конфигурации администратору;
- пользователю должно отображаться понятное сообщение о технической ошибке.

9. Режимы обработки

Поддержать режимы обработки через пресеты.

Минимум 3 пресета:
1. catalog
   - очень аккуратная обработка;
   - минимальные вмешательства;
   - подходит для карточек товаров и нейтральной подачи.

2. soft
   - мягкое улучшение;
   - немного более заметная яркость и контраст;
   - режим по умолчанию.

3. juicy
   - чуть более выразительная подача;
   - чуть более насыщенное и контрастное изображение;
   - при этом без агрессивной обработки.

Каждый пресет должен задавать:
- пределы коррекции яркости;
- пределы коррекции контраста;
- пределы коррекции насыщенности;
- силу denoise;
- силу sharpening;
- параметры локального контраста;
- параметры наложения логотипов при необходимости.

10. Формирование результата

10.1. Выходной формат
На MVP:
- сохранить в JPEG или оставить исходный формат, если это допустимо;
- предпочтительно:
  - JPG/JPEG → JPG;
  - PNG → PNG либо JPG по конфигу;
  - WEBP → WEBP либо JPG по конфигу.

Рекомендуется сделать это настраиваемым.

10.2. Качество сохранения
Сделать настраиваемыми:
- JPEG quality;
- PNG compression;
- WEBP quality.

Система не должна чрезмерно пережимать изображения.

10.3. Имена файлов
Имена файлов должны сохраняться.
Если есть риск конфликтов имен:
- добавлять безопасный суффикс;
- логика должна быть описана в коде и конфиге.

10.4. Варианты выдачи результата
Поддержать:
- итоговый ZIP-архив;
- итоговую папку на Яндекс Диске.

По умолчанию на MVP рекомендуется:
- загружать итоговый ZIP-архив.

11. Интеграция с Яндекс Диском

11.1. Вход
Сервис должен уметь:
- принимать публичную ссылку;
- получать прямой URL на скачивание через API Яндекс Диска или соответствующий механизм;
- скачивать архив по этой ссылке.

11.2. Выход
Сервис должен уметь:
- загружать итоговый файл или папку на служебный Яндекс Диск;
- создавать публичную ссылку;
- возвращать ее пользователю в Telegram.

11.3. Структура хранения результата
Рекомендуемая структура на Яндекс Диске:
- /processed/{YYYY-MM-DD}/{job_id}/result.zip
или
- /processed/{YYYY-MM-DD}/{job_id}/files/

11.4. Срок хранения
Предусмотреть политику хранения:
- например, автоматическое хранение результатов 7, 14 или 30 дней;
- значение должно задаваться в конфиге;
- очистка старых результатов должна выполняться отдельным служебным процессом.

12. Отчет по задаче

После завершения бот должен отправлять:
- статус: завершено / завершено с предупреждениями / ошибка;
- job_id;
- число найденных файлов;
- число обработанных изображений;
- число пропущенных файлов;
- ссылку на результат.

Дополнительно желательно:
- прикладывать текстовый отчет report.txt или report.json к задаче на Яндекс Диске.

В отчете по каждому пропущенному файлу указывать причину:
- неподдерживаемый формат;
- поврежденный файл;
- ошибка чтения;
- слишком маленькое изображение;
- ошибка наложения логотипа;
- иное.

13. Логирование и мониторинг

13.1. Логирование
Нужно логировать:
- создание задачи;
- входную ссылку;
- этапы выполнения;
- ошибки скачивания;
- ошибки распаковки;
- ошибки обработки отдельных файлов;
- ошибки загрузки результата;
- время выполнения этапов;
- итоговые метрики.

13.2. Технические логи
Логи должны быть разделены на:
- системные;
- по задачам;
- по файлам.

13.3. Уровни логов
Поддержать:
- INFO
- WARNING
- ERROR

14. Требования к производительности

Для MVP система должна:
- корректно обрабатывать архив до 5 ГБ;
- корректно обрабатывать до 150 изображений на одну задачу;
- не использовать загрузку всего архива в RAM;
- не использовать загрузку всех изображений в RAM одновременно;
- поддерживать ограниченную параллельность.

15. Требования к инфраструктуре

Минимально рекомендуемая конфигурация для MVP:
- 4–8 vCPU;
- 8–16 ГБ RAM;
- SSD;
- свободное место на диске с запасом минимум 30–50 ГБ.

Важно:
На одну активную задачу с архивом 5 ГБ нужно закладывать запас диска под:
- входной архив;
- распакованный архив;
- выходные файлы;
- временные служебные файлы.

16. Очистка временных данных

После завершения задачи система должна:
- удалить временный архив;
- удалить распакованные исходники;
- удалить локальные результаты после успешной загрузки на Яндекс Диск;
- при ошибке сохранять временные данные ограниченное время только для диагностики, если это включено конфигом.

17. Обработка ошибок

Предусмотреть обработку ошибок:
- ссылка невалидна;
- ссылка недоступна;
- файл не ZIP;
- архив поврежден;
- архив слишком большой;
- в архиве нет поддерживаемых изображений;
- ошибка скачивания;
- ошибка распаковки;
- ошибка обработки одного или нескольких изображений;
- ошибка загрузки результата на Яндекс Диск;
- ошибка публикации ссылки;
- отсутствие логотипов;
- нехватка места на диске;
- отмена задачи пользователем.

Если ошибка возникла на одном файле:
- задача в целом может быть завершена;
- файл отмечается как пропущенный;
- причина попадает в отчет.

Если ошибка критическая:
- задача переводится в failed;
- пользователю отправляется понятное сообщение.

18. Конфигурируемые параметры

Все основные параметры должны задаваться через конфиг:
- лимиты размера архива;
- лимиты количества файлов;
- поддерживаемые расширения;
- число параллельных воркеров;
- временные директории;
- параметры улучшения изображения;
- пресеты обработки;
- параметры логотипов;
- параметры загрузки на Яндекс Диск;
- политика хранения результатов;
- настройки логирования;
- лимиты времени выполнения задачи.

20. Административные требования

Администратор должен иметь возможность:
- менять логотипы без правки кода;
- менять прозрачность и размер логотипов;
- менять пресеты обработки;
- менять лимиты;
- включать и выключать режим публикации результата;
- задавать папку на Яндекс Диске для сохранения результатов.

21. Требования к безопасности

- не исполнять содержимое архива;
- проверять пути при распаковке, чтобы избежать path traversal;
- ограничивать допустимые типы файлов;
- валидировать размер входного архива;
- ограничивать размер распакованных данных;
- не хранить публичные ссылки дольше необходимого;
- использовать отдельные служебные токены для Telegram и Яндекс Диска;
- не логировать секреты в открытом виде.

22. Что должно быть реализовано в MVP

Обязательно:
1. Telegram-бот.
2. Прием публичной ссылки Яндекс Диска на ZIP-архив.
3. Скачивание архива потоком.
4. Распаковка.
5. Обработка JPG/JPEG/PNG/WEBP.
6. Мягкое улучшение изображения через adaptive filter pipeline.
7. Наложение двух полупрозрачных логотипов:
   - нижний левый угол;
   - нижний правый угол.
8. Сохранение результата.
9. Сборка итогового ZIP.
10. Загрузка ZIP на Яндекс Диск.
11. Создание публичной ссылки.
12. Возврат ссылки и отчета через Telegram.
13. Статусы задач.
14. Логи и базовый отчет по ошибкам.

Желательно, но не обязательно для первой версии:
- выбор пресета кнопками в боте;
- загрузка результата не только ZIP, но и папкой;
- отдельная админ-панель;
- превью 2–3 тестовых изображений до запуска полной обработки.

23. Критерии приемки

Система считается принятой, если:
1. Пользователь может отправить публичную ссылку Яндекс Диска в Telegram-бот.
2. Система корректно скачивает ZIP-архив.
3. Система корректно распаковывает архив.
4. Система обрабатывает поддерживаемые изображения.
5. На каждое обработанное изображение накладываются два полупрозрачных логотипа в нижних углах.
6. Результат загружается на Яндекс Диск.
7. Пользователь получает публичную ссылку на итоговый результат.
8. При частичных ошибках система формирует отчет.
9. При критических ошибках система корректно сообщает о проблеме.
10. После завершения временные данные очищаются.

24. Дополнительные пожелания на следующую итерацию

На следующую версию можно предусмотреть:
- web-интерфейс вместо или вместе с Telegram-ботом;
- поддержку нескольких комплектов логотипов;
- выбор положения логотипов по пресету;
- предпросмотр результата;
- сравнение до/после;
- поддержку RAW через отдельный модуль;
- пакетную работу с несколькими архивами;
- уведомления о завершении в отдельный Telegram-чат;
- SLA и лимиты по пользователям.
